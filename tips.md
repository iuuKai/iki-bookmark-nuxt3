# iki-bookmark-nuxt3 项目评估（技术负责人 + 面试官视角）

## 一句话结论

这是一个“产品闭环完整、工程实现扎实、性能优化有实践深度”的中高质量个人项目，明显强于只做 CRUD 的展示型项目。

## 总体评分（10 分制）

- 产品完整度：8.5
- 工程设计：8.0
- 性能优化能力：8.5
- 可维护性：7.5
- 代码规范与稳定性：7.5
- 面试说服力：9.0

## 我会认可的亮点

1. 业务闭环完整  
从 OAuth 登录、仓库数据读写、书签管理到展示与优化，全链路可运行，具备真实业务价值。

2. 数据 ownership 设计清晰  
把数据存储在用户仓库，降低中心化后端依赖，项目思路有产品和架构意识。

3. 有“性能问题 -> 定位 -> 收敛”的实战  
不是泛泛说“做了懒加载”，而是具体处理了：
- 图片并发过高导致主线程压力；
- 菜单动画在批量 icon loading 时掉帧；
- 首屏骨架态和真实数据渲染时序问题。

4. 前后端一体化落地能力  
Nuxt/Nitro + 前端组件 + store + server API 协作顺畅，说明具备端到端交付能力。

## 我会追问和扣分的点

1. 类型与 lint 稳定性  
出现过隐式 any、规则告警、配置兼容问题。虽然后续都修了，但说明基础工程治理还可前置。

2. 配置与依赖版本治理  
TS/Stylelint/插件版本在升级时暴露兼容问题，建议形成固定升级策略和 CI 校验链。

3. 观测能力不足  
优化过程如果没有明确指标（如 LCP、INP、首屏节点数、请求并发曲线），面试时说服力会打折。

4. 异常路径可继续加强  
请求层、代理失败、图标降级虽有处理，但可进一步补充错误码分层和可观测日志。

## 面试官最关心的 6 个问题（建议答案方向）

1. 你如何证明优化有效？  
给出优化前后数据：首屏渲染时间、菜单交互帧率、并发请求数、内存峰值。

2. 为什么不直接上第三方 favicon 服务？  
从可控性、隐私、稳定性、外部依赖风险回答，并给出你当前回退策略。

3. 你如何避免骨架态卡死？  
强调请求层必须“有终态”（resolve/reject），并说明超时和错误兜底路径。

4. 菜单动画为什么会被 icon 加载拖慢？  
解释主线程竞争、重绘区域扩大、批量动画导致帧预算被抢占。

5. 你做了哪些“低风险高收益”优化？  
并发队列、静态占位、视区渲染、简化 tooltip 实例、transform 动画替代重排动画。

6. 后续如何演进？  
虚拟滚动、图标离线预处理、性能埋点、CI 质量门禁。

## 给你的提升建议（高优先级）

1. 建立性能基线文档  
至少记录 3 项指标：首屏时间、可交互时间、页面内动画帧率。

2. 补一套轻量 CI  
`type-check + lint + build`，避免“改一点出一串告警”。

3. 增加关键路径注释和 ADR  
对图标策略、并发队列、骨架时序做简短设计说明，方便面试讲解。

4. 用一次真实压测数据做收尾  
例如 500/1000 个书签下的加载和滚动表现，对简历非常加分。

## 面试结论（如果我是面试官）

这项目可以作为“前端中级到中高级候选人”的主项目讲解，尤其适合突出你在“复杂页面性能和工程稳定性”上的实践。  
如果你把指标化验证和 CI 治理再补上，整体竞争力会再上一个台阶。
